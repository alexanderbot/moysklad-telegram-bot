## Архитектурные улучшения

### 4. ~~Несогласованность часовых поясов в get_period_dates()~~ ✅
**Проблема:** Функция `get_period_dates()` в `moysklad_api.py` использует `datetime.now().date()` вместо `today_moscow()` из `config.py`. Если сервер в другом часовом поясе — все границы периодов будут неверны.

**Решение:** Заменить `datetime.now().date()` на `today_moscow()` (импорт из `config`).

**Файлы:** `moysklad_api.py` — **выполнено:** добавлен импорт `today_moscow`, базовая дата периода берётся через `today_moscow()`.

### 5. Дублирование логики пагинации
**Проблема:** Цикл пагинации (while + offset + limit + защита от 100000) повторяется 6 раз в `moysklad_api.py`.

**Решение:** Вынести в метод `_paginated_request()`:
```python
async def _paginated_request(self, endpoint: str, params: dict) -> list[dict]:
    results = []
    offset = 0
    limit = params.get("limit", 1000)
    while True:
        params["offset"] = offset
        page = await self._make_request(endpoint, params)
        if not page or "rows" not in page:
            break
        rows = page["rows"]
        results.extend(rows)
        if len(rows) < limit:
            break
        offset += limit
        if offset > 100_000:
            break
    return results
```

**Файлы:** `moysklad_api.py`

### 6. Отсутствие кэширования API-запросов
**Проблема:** Каждый запрос быстрого отчета делает 6 обращений к API МойСклад. При частых запросах от одного пользователя — лишняя нагрузка.

**Решение:** Добавить кэш с TTL 5–10 минут (например, `cachetools.TTLCache` или простой dict с timestamp).

**Файлы:** `moysklad_api.py`

---

### 7. Отсутствие retry-логики
**Проблема:** При ошибках 429 (rate limit) или таймаутах запросы не повторяются.

**Решение:** Добавить `tenacity` или ручной retry с exponential backoff в `_make_request()`.

**Файлы:** `moysklad_api.py`

## Новые бизнес-метрики

### Приоритет 1 — Прибыль и маржинальность
- API МойСклад: `report/profit/byproduct`, `report/profit/byvariant`
- Валовая прибыль за день / неделю / месяц
- Маржинальность (%) по периодам
- Самые маржинальные и убыточные товары

### Приоритет 2 — Остатки на складе + алерты
- Метод `get_stock_report()` уже написан, но не подключён к обработчикам
- API: `report/stock/all`
- Текущие остатки по складам
- Товары с критически низким остатком
- Неликвиды (давно не продавались)
- Прогноз, на сколько дней хватит запасов
- Стоимость остатков (замороженные деньги)

### Приоритет 3 — Выполнение плана продаж
- Пользователь задаёт цель на месяц
- Бот показывает прогресс-бар и прогноз выполнения
- Необходимая дневная выручка для достижения плана

### Приоритет 4 — Дебиторка / Денежный поток
- API: `entity/paymentin`, `entity/paymentout`
- Входящие и исходящие платежи за период
- Чистый денежный поток
- Дебиторская задолженность (кто должен вам)
- Кредиторская задолженность (кому должны вы)

### Приоритет 5 — Топ клиентов
- API: `entity/counterparty`
- Топ-10 покупателей по выручке
- Новые клиенты за период
- Потерянные клиенты (не покупали > 30/60/90 дней)

### Приоритет 6 — ABC-анализ товаров
- Категория A: 20% товаров = 80% выручки (фокус)
- Категория B: 30% товаров = 15% выручки
- Категория C: 50% товаров = 5% выручки (кандидаты на вывод)

### Приоритет 7 — Конверсия воронки продаж
- Заказ → Отгрузка → Оплата: % перехода на каждый этап
- Средний срок от заказа до отгрузки

- Крупный заказ (сумма выше порога)
- Товар заканчивается на складе
- Выручка за день ниже среднего на 30%+
- Рекордная дневная выручка

### Приоритет 9 — Экспорт отчетов в Excel/PDF
- Отправка `.xlsx` файлов через Telegram `send_document`
- Формирование через `openpyxl`

### Приоритет 10 — Продажи по дням недели / часам
